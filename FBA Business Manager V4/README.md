> **Open architecture question the user raised (answered, not yet built):** where inventory UNITS are deducted. Answer = **neither Production nor the invoice**. Production lines = the *plan* (ordered qty + expected cost). Invoice goods lines = the *actual money* (billed), joined to Production by SKU only for reconciliation + landed-cost. So there is no double-deduction — units flow Production(ordered) → Shipment(shipped) → FBA(received=inventory in); the invoice is a parallel cost track keyed by SKU, never a unit transaction.

## What this is

An interactive HTML/React prototype of **Vyonix Business OS** — an operations platform for an Amazon FBA / import business. It covers the lifecycle of a purchase order: production → shipping → inspection → invoices → closeout.

Rendered as standalone HTML files using **React 18 + Babel (in-browser, no build step)**. Each `.jsx` file attaches its components to `window` so later scripts can use them. Shared styling lives in one stylesheet (`vy-app.css`).

### Shared building blocks (used by all entry points)
- **`vy-icons.jsx`** — `<VyIcon name=… size=… />` icon set. Check here for valid icon names before using one.
- **`vy-shell.jsx`** — App chrome: `VySidebar` (grouped left nav), `VyHeader` (search + theme + actions), `VyMobileNav`, **`VyEmptyState`** (shared first-run / no-results block — dashed card, icon chip, title, body, action buttons; props `icon`/`title`/`body`/`actions[{label,icon,onClick,href,primary}]`/`tone`; used by the Orders + Catalog lists). **v1 nav is trimmed to only built pages:** two groups — **Operations** (Orders · Invoices · Shipments · FBA Shipments) and **Catalog** (Products · Inventory · Suppliers · Partners); every item has a working `href`. (Removed for v1: Command/Dashboard, the Amazon group, Money, Marketing, Business, Quote Requests, Suppliers/Partners/Contacts — all unbuilt placeholders or duplicates. App lands on Orders.) **`VySidebar`/`VyMobileNav` take an optional `active` prop** (nav-item label) that overrides static active flags + opens the owning group. **`VyHeader`** no longer renders the "Workspace · Operations" label + section pills (removed as redundant with the sidebar) — it still accepts `workspaceName`/`tabs`/`activeTab` props for backward-compat but ignores them; `OPERATIONS_TABS`/`CATALOG_TABS` remain exported but unused. Search sits left, actions (bell/theme/avatar) right. **The search button opens the global command palette** (`window.vyOpenSearch()` from `global-search.jsx`; falls back to the page's `onOpenSearch` prop if the palette isn't loaded).
- **`image-slot.js`** — `<image-slot>` web component (starter). User-fillable image placeholder; drops persist via a sidecar. Used by the Product page gallery. Plain JS — load with `<script src>` (NOT babel).
- **`vy-overlays.jsx`** — Shared overlays: activity drawer, sheets, dialogs, more-menu.
- **`global-search.jsx`** — **⌘K global command palette ("Search anything").** Self-mounting (appends its own `#vy-global-search-root` to `<body>` and renders with `ReactDOM.createRoot` — no per-page React wiring), so every page just loads the script after its data files. Opens on **⌘K / Ctrl+K**, on the `vy-open-search` event, or via `window.vyOpenSearch()` (the `VyHeader` search button calls this). **Builds its index lazily on each open, feature-detecting whichever data sources the page loaded:** Orders (`ordAllOrders`→`?order=`), Products + their variant SKUs (`catLoadFamilies`/`CAT_FAMILIES`→`Vyonix Product.html?family=`), Invoices (`PAY_INVOICES`→`Order Shell ?order=#invoices`), and (when present) Suppliers/Partners/Shipments. Token-AND substring match with light relevance scoring, grouped results, full keyboard nav (↑/↓/Enter/Esc), click-outside to close. Styling lives in `vy-app.css` (`.vgs-*`) per convention — **keyframes animate transform only (no opacity)** so the panel is never stuck invisible under reduced-motion. Loaded on every app page.
- **`vy-app.css`** — **The single source of truth for all styling.** HSL CSS variables, light/dark themes, every `vy-*` class. Edit styling here, not inline.
- **`fba-calc.jsx`** — **Forward-looking FBA sourcing calculator.** Exposes `FbaCalculator` (the body), `FbaCalculatorPage` (standalone page = chrome + body, mounted inline by `Vyonix FBA Calculator.html`), and a self-mounting **modal root** (`#vy-fba-calc-root`) that registers `window.vyOpenFbaCalc(seed)`. Inputs: target sale price, landed-cost **buildup** (unit cost + freight/unit + duty%) or single figure, dims (L/W/H) + weight, category (referral % preset, editable), storage months + Q4-peak toggle, order quantity. Outputs: net/unit, margin %, ROI %, **breakeven sale price**, per-unit waterfall, and an order-size projection (cash in goods / revenue / projected profit). A verdict badge (Strong/Healthy/Thin/Loss) reads off margin+ROI. **The fee model (`fcSizeTier`/`fcFulfillmentFee`/`fcReferralPct`/`fcStorageMo`) is an intentional copy of product-app.jsx's so a SKU reconciles whether viewed here or on its Product page.** Automotive catalog categories are pinned to 15% to match. Prefill: `catLoadFamilies()`/`CAT_FAMILIES` (guarded — picker hides if absent) via an in-calc variant picker, the page's `?sku=` param, or a `seed` object passed to `vyOpenFbaCalc` (the Product variant drawer's "Model a reorder" button does this). The modal is keyed per-open so reopening with a new SKU resets all fields. All hooks/consts are `fc`-prefixed to avoid the shared-global-scope collisions the Babel setup is prone to.

### Per-screen files
- **`inventory-data.jsx`** — Inventory layer. **Flattens `CAT_FAMILIES` variants into per-SKU rows** (`INV_ROWS`) and attaches live-stock fields: `onHand`/`reserved`/`unfulfillable`/`inbound`/`velocity` (Amazon), `reorderPoint` (Manual), `fc` (round-robin across `INV_FCS`). This is where **Catalog (master) and Inventory (live state) connect — join key SKU→FNSKU**. `invStats(row)` derives `available` (= onHand − reserved − unfulfillable), `daysCover`, and `health` (Healthy/Low/Reorder vs the reorder point). `INV_HEALTH_TONE`, `INV_SAFETY_DAYS`, `INV_LAST_SYNC`. **Must load after `catalog-data.jsx`.**
- **`inventory-app.jsx`** — `InventoryPage` (mounts on `Vyonix Inventory.html`). Header (Export + Sync from Amazon) → 5 KPIs (On hand · Available · Inbound · Reorder now · Unfulfillable) → **Amazon sync strip** ("Synced from Seller Central · FBA Inventory API") → filter bar (search + Category/Supplier + health chips + **Edit reorder pts** toggle) → per-SKU table (SKU links to its Product page; On-hand/Reserved/Avail/Inbound/Days-cover/Reorder-pt/Status; per-row **Reorder** for non-healthy rows → opens **Orders List** with the create-order sheet **pre-seeded** with that SKU + a suggested qty (velocity × (lead time + 45d), net of available + inbound, rounded to 25s, floor 50) + supplier + last unit cost, passed via URL params (`?reorder=1&sku=…&qty=…`)) → **By fulfillment center** distribution bars. `InvSourceTag` (Amazon/Manual) on Amazon-synced vs manual columns. Stock levels in `catalog-data.jsx` are tuned so the catalog reads healthy with the microfiber-cover family (`msc`) as the clear reorder story (12 Healthy / 6 Reorder).
- **`catalog-data.jsx`** — Catalog dataset (`CAT_FAMILIES`: product families → variants enriched with **FNSKU/ASIN** (Amazon), **dims/weight/brand** (Amazon), FBA stock + inbound (Amazon), last cost ¥/$ + supplier/lead-time/MOQ (Manual), plus per-family `costHistory` and `orderHistory`). Helpers: `catFamilyById`, `catFamilyStats` (skuCount/stock/inbound/cost-range/gaps/reorder/**health**), `CAT_LOW_STOCK`, `CAT_HEALTH_TONE`. **Shared persistence (localStorage `vy_catalog_families_v1`):** `catLoadFamilies()` / `catSaveFamilies()` / `catUpdateFamilyVariants(id, variants)` / `catResetFamilies()` — created products + variant edits survive reloads and are **shared across the Catalog list, Product page, and Inventory** (all read `catLoadFamilies()`; `CAT_FAMILIES` is just the seed). Named `CAT_*` to avoid colliding with the add-skus modal's `CATALOG_FAMILIES`.
- **`catalog-app.jsx`** — `CatalogListPage` (mounts on `Vyonix Catalog.html`). Calm browse-and-filter: page header (Export + New product) → 4 icon KPIs (Products · Variants · Reorder · Data gaps) → filter bar (search by product/SKU/FNSKU/ASIN/supplier + Category/Supplier selects + health chips) → one contained card of **family rows** (striped thumb, title+health badge, meta, FBA stock, cost range). Row click → `Vyonix Product.html?family=<id>`. **`families` is state** — the **New product** button opens `CatNewProductModal` (product fields + repeatable variant rows with SKU/name/pack/cost); creating prepends the family (badged **New**, FNSKU/ASIN "Pending sync", 0 stock) and clears filters so it shows at the top. (New products live in the list session only — cross-page state isn't shared, so opening one falls back to the first product.)
- **`product-app.jsx`** — `ProductPage` (mounts on `Vyonix Product.html`, reads `?family=`; **loads the family from the shared store via `catLoadFamilies()`** so products created on the list open correctly). Single-column stack: breadcrumb → header (title, category/brand/supplier chips, health badge, Reorder/Edit/More) → **Images** (3 `<image-slot>` — user drops real shots, persist) + facts grid → **Variants** table (SKU · variant · FNSKU · FBA · cost $/¥ · status; **Add variant** modal `ProdAddVariantModal` appends a variant, **Edit cost** toggles inline ¥/$ inputs — both persist via `catUpdateFamilyVariants`) → **Details** (specs+supplier field grids) → **Cost history** (bars) → **Order history** (links to `Vyonix Order Shell.html?order=`). `ProdSourceTag` (Amazon/Manual) reused across facts, variant headers, and detail fields.
- **`orders-app.jsx`** + **`orders-data.jsx`** — Orders list page logic + its sample data. **`orders-data.jsx` also owns the shared draft-order store** (localStorage `vy_orders_drafts_v1`): `ordLoadDrafts()` / `ordSaveDrafts()` / `ordAddDraft(draft)` / `ordClearDrafts()` / `ordAllOrders()` (drafts first, then seed `ORDERS_LIST`, deduped by id). Draft rows survive reload. (The Order Shell still renders fixed mock data and doesn't read `?order=`, so a draft row opens the canned shell — same limitation as every other row.) **Order STATUS is now live + persisted** — `ORD_PIPELINE` (Draft→In production→Inspection→In transit→At FBA→Closed) + `ordAdvanceStatus`/`ordStepBackStatus`/`ordResetStatus`/`ordStatusKey`/`ordStageForStatus` (localStorage `vy_order_status_v1`); `ordAllOrders()` overlays the saved status (status/tone/nextStep) onto every row, so advancing an order on its shell reflects in the list and survives reload. **Data-driven advance:** `ordAdvanceToAtLeast(id, seedLabel, targetKey, reason)` + `ordRankOf` move an order **forward only** to a section-milestone stage (never regressing a manual step-back), persist + dispatch `vy-order-status-changed`, and surface `vyFlashStatus(msg)` (an imperative bottom-center toast). Helpers: `LOG_ORDER_SCOPE` (per-order committed units), `LOG_ORDERS` (order directory for the New-shipment form), `LOG_STAGES`, `LOG_FCS`, tone maps (`LOG_STAGE_TONE`/`LOG_CUSTOMS_TONE`/`LOG_FBA_TONE`), `logShipFbaStats(s)` (rolls a shipment's inbounds → expected/received/variance/short), `logAllFbaRows()` (flattens every inbound into a row carrying parent shipment + order). **Draft-shipment store** (localStorage `vy_shipments_drafts_v1`): `logLoadDrafts`/`logSaveDrafts`/`logAddDraft`/`logClearDrafts`/`logAllShipments` (drafts first + seed, dedupe by id) + `logPackedForOrder(orderId)` (for the unpacked-remainder suggestion). One order (`ORD-2026-05-008`, 500 u) is intentionally **unshipped** so the funnel reads ~92% and New-shipment has a remainder to suggest. **Reconciliation chain ordered → packed → shipped → received holds across both pages.**

Header (Export + **New shipment**) → 5 KPIs (Shipments · On the water · Packed % · Received · Freight $) → **Pipeline reconciliation funnel** (`ShpFunnel`: ordered→packed→shipped→received bars, Manual/Amazon source tags) → filter (search + Mode + Forwarder + stage chips) → table (Shipment/mode/BOL · Order→Order Shell · Route · Forwarder/incoterm · ETD→ETA · Cargo CBM/kg/ctn · Packed · **FBA inbounds count → FBA page filtered by `?shipment=`** · Customs · Status). The Shipments-list row title (`.vy-row-title` on the shipment ID) links here; the `ShpDrawer` is the quick resume (its footer now has **"Open full shipment"** primary). Header (Export + Sync) → 5 KPIs → sync strip → unlinked-orphan banner → filter → table → by-FC distribution. **Row title links to `Vyonix FBA Shipment.html?fba=<id>` (full page); row click opens the quick `FbaDrawer`** — same drawer+page pattern as the physical Shipments list. `FbaLinkModal` attaches unlinked inbounds to a shipment/order (`logSaveFbaLink`).

### Cross-section data flow (important)
Sections hold their own state, but a few **one-way bridges** keep numbers consistent. These are **in-memory** — they work as you navigate tabs in a session, but a hard reload that lands directly on a downstream section won't see them (the upstream section hasn't mounted to publish).
- **Invoices → Closeout:** `coDeriveBuckets()` reads `window.INV_INVOICES`, walks `payInvoiceLines(inv)` and buckets each service line by its **charge type** (`CO_BUCKET_FOR_CT`); goods use the **billed** per-SKU amount (`coGoodsBilledMap` ← `paySupplierGoods`).
- **Production → Closeout:** user-added **inventoriable** non-product costs publish to `window.__VY_PROD_EXTRA_COSTS` (Production useEffect); Closeout folds them in as "Other inventoriable" bucket(s) via `coExtraBuckets()`, allocated by the chosen basis. Period-expense costs are excluded.
- **Production / Closeout landed cost** use the same allocation model (fees by value; freight/inspection by units) so the EST LANDED / U column reconciles with Closeout's per-SKU figures.
- **Sections → order status (data-driven):** section milestones auto-advance the order's lifecycle status via `ordAdvanceToAtLeast(id, seedLabel, targetKey, reason)` (in `orders-data.jsx`) — **Inspection** clears (report accepted + Pass) → ≥ `transit`; **Shipping** all FBA inbounds received → ≥ `fba`; **Closeout** locked **and** every bill settled → `closed`. The helper is **forward-only** (it never regresses a manually-advanced order) and each section fires it on the **rising edge** of its milestone (a `React.useRef` guard, so it never re-fires on mount for an order that loaded already-past the milestone). The manual **Advance / Step-back / Reset** controls on the Home journey remain the override. On advance it writes the status (persisted, `vy_order_status_v1`), shows a brief imperative toast (`vyFlashStatus`), and dispatches a `vy-order-status-changed` event; the Home **header badge + journey** listen for it (and re-read live status on mount) so the new stage reflects without a reload. Gated on `window.VY_CURRENT_ORDER.id`, so it's inert on the standalone section pages.

## Hybrid data model (manual + Amazon SP-API)

The order lifecycle spans two data worlds: the **upstream** half (Production, Inspection, supplier Invoices/payments, freight) is inherently **manual** — none of it exists in Seller Central — while the **downstream** half (FBA inbound status, received units, inventory, FNSKU/dims, Amazon fees) can be **synced from Amazon SP-API**. The app is built **hybrid**: keep upstream manual-but-structured, let the API auto-fill only a small, high-value slice (inbound status, received qty, inventory, dims). Pull *less* on purpose to stay simple.

To make this visible, fields carry a **source marker**: `ShipSourceTag` (in `shipping-app.jsx`) renders a quiet uppercase tag — **● Amazon** (filled info dot = synced from Seller Central) vs **○ Manual** (hollow dot = entered by you). **Reference example:** the **Shipping → FBA inbound** card — *Expected* is tagged Manual (your packing allocation), *Received* is tagged Amazon (synced), the card subtitle states "Status & received units sync from Seller Central," and each row shows "Amazon · synced ‹time›" with a refresh glyph. Roll this same `source="amazon"|"manual"` pattern across the other sections as the hybrid model is wired in.

> **Conceptual note — Shipment vs FBA Shipment (discussed 2026-05-31).** These are two different objects and are deliberately kept separate.
